# baseline 重构:schema DB 物化 + snapshot 差异化 SQL 产物

> 来自 proposal: proposals/20260512152112-baseline-snapshot-diff/

## 目标

`drizzleman baseline --empty-schema-db-url=<url>` 改成「target DB introspect + 本地 schema 推到 schema DB + schema DB introspect + 两份 snapshot 结构化差异 → 直接生成 0001_delta.sql 与 schema.sql」流程,完全干掉对本地 TS 文件的 regex 解析。

**术语**:
- **target DB** — 从 `drizzle.config.ts` 的 `dbCredentials` 解析出的真实库。
- **schema DB** — 用户额外提供、起始为空的库,仅用作「物化本地 schema → 出 snapshot」的容器。

## 改动范围

- **新增**:
  - `src/db/pg.ts` 中加 `assertSchemaDbEmpty(creds)`:连到 schema DB url,枚举非系统 schema 下的表 / 视图;非空抛错并附带前 N 个具体名字。dispatcher 在 `src/db/index.ts` 加对应入口;mysql/sqlite 实现暂以 `not implemented` 抛错,但保留方法签名占位。
  - `src/hooks/baseline.ts` 中加:
    - `--empty-schema-db-url=<url>` flag(以及 `DRIZZLEMAN_EMPTY_SCHEMA_DB_URL` 环境变量后备);缺失即报错退出。
    - Step B(新):assert schema DB empty。
    - Step C(新):drizzle-kit push to schema DB url(显式 `--dialect` / `--schema` / `--url`,继续走 `passthrough`)。
    - Step D(新):drizzle-kit introspect schema DB url → 一个新建的 `tmpDir`(放在 outDir 下,后缀 `.baseline-schemadbintro-<ts>`,跑完即删,不参与 promote)。
    - Step E:读两份 `meta/0000_snapshot.json`,做结构化 diff(tables / columns / indexes / foreignKeys / enums),输出 5 类差集:
      - `onlyInSchemaDb.{tables,columns,indexes,foreignKeys,enums}`
      - `onlyInTarget.{tables,columns,indexes,foreignKeys,enums}`
    - Step F:解析两份 0000.sql 的纯文本,把语句切成「(kind, key, sqlText)」三元组列表(kind ∈ enum / table / index / fk;key 用稳定标识,如 `table:public.users` 或 `index:public.users_email_idx`);regex 在 introspect 输出上跑,格式可控。
    - Step G:按差集筛三元组,按原文件出现顺序拼装:
      - 写入 `previewDir/0001_delta.sql`(从 schema DB 三元组里挑 `onlyInSchemaDb` 的)。
      - 写入 `previewDir/schema.sql`(从 target 三元组里挑 `onlyInTarget` 的);空时写注释占位 `-- schema.sql\n-- (generated by drizzleman baseline)\n-- target DB structure matches local schema; nothing to add.\n`。
    - Step H:删 `tmpDir`;preview 段重写,按 4 个文件 + diff 统计(每类差几个、总条数)展示;decision prompt 保留(已实现的 yes/TTY 路径不变)。
    - 收尾打印「schema DB `<url>` 已被 push 进本地 schema,请自行 drop / 回收」。
- **更新**:
  - `README.md`:把 baseline 那一行刷新成新流程描述(必填 `--empty-schema-db-url`,产出 4 份文件,schema.sql 必有,提示用户清理 schema DB)。
  - `src/types.ts` / `src/config.ts` 不动(schema 字段已加过)。
- **删除**:
  - `src/hooks/baseline.ts` 里 `extractDrizzleTables` / `listLocalSchemaFiles` / `buildSchemaDiff` / 旧「Schema diff (Drizzle DSL)」预览段,以及之前 SQL summary 里的 dropped/added 那块(被 snapshot diff 替代)。
  - 不删 `summarizeDiff` 也可以保留作为辅助,但目前没新用途 → 一并删,减少死代码。

## 关键点

- **schema DB 空状态校验必须在 push 之前且独立成步**:绝不能让 push 自己去碰非空库 —— 一旦 push 出错回滚不彻底,会污染用户给的那个库。先 query → 再 push。
- **SQL 切块的 regex 在 introspect 输出上跑**:introspect 输出格式稳定(每条语句以 `;` 收尾、表名带双引号、ALTER 单独成行)。**绝不**对用户写的本地 `.ts` 文件做正则;那是上一版的坑,这一版核心就是绕开它。
- **entity key 设计**:必须能在 snapshot JSON 与 SQL 文本两边对齐。建议:
  - `enum:<schema>.<name>` ↔ `CREATE TYPE "schema"."name" AS ENUM (...)`(无 schema 时归到 `public` 或与 snapshot 保持一致)
  - `table:<schema>.<name>` ↔ `CREATE TABLE "schema"."name" (...)`
  - `index:<schema>.<name>` ↔ `CREATE [UNIQUE] INDEX "name" ON ...`(index 名全局唯一,schema 跟随所属表)
  - `fk:<schema>.<table>.<constraint>` ↔ `ALTER TABLE ... ADD CONSTRAINT "name" FOREIGN KEY ...`
  - `column:<schema>.<table>.<col>` —— 这一类**没有独立 SQL 语句**(列在 `CREATE TABLE` 里内联),处理方式:
    - 若 `onlyInSchemaDb.columns` 里某列对应的表本身已经在 `onlyInSchemaDb.tables` 里 → 跳过(整张表的 CREATE 已经覆盖了)。
    - 否则该列要单独写 `ALTER TABLE ... ADD COLUMN <col> <type> [DEFAULT ...] [NOT NULL]` —— 类型/默认值/可空性从 snapshot 里取(snapshot 字段足够构造)。schema.sql 同理。
- **drizzle-kit push 的 flag 模式**:同 introspect/generate,`--config` 不能和其它 flag 混用 → 全部展开为显式 flag(`--dialect=` + `--schema=` + `--url=`)。
- **tmpDir 命名与清理**:用 `.baseline-schemadbintro-<ts>/` 前缀,与 `.baseline-preview-` 区分;成功结束必删,中途失败留下方便排查。
- **打印 url 时遮蔽密码**:复用 `src/url.ts` 的 `targetUrl` 思路 / `stripPassword`;schema DB url 也走一遍。
- **错误恢复**:push 失败 → schema DB 可能半残;明确告知用户,引导他们 drop & retry。我们不尝试自动回滚 push。

### 做了什么

- **`src/db/pg.ts`** — 新增 `assertSchemaDbEmpty(creds)`:连到 schema DB,查 `information_schema.tables` 排除 `pg_catalog` / `information_schema` / `pg_toast*` / `pg_temp_*` / `pg_toast_temp_*`,>0 抛错并附带前 20 个表名作为非空证据。
- **`src/db/mysql.ts`** / **`src/db/sqlite.ts`** — 各加 `assertSchemaDbEmpty` stub,抛 `not implemented` 占位;签名与 pg 对齐。
- **`src/db/index.ts`** — 加 `assertSchemaDbEmpty` dispatcher,按 dialect 转发。
- **`src/hooks/baseline.ts`** — 整体重写:
  - 删除旧 `extractDrizzleTables` / `listLocalSchemaFiles` / `buildSchemaDiff` / `summarizeDiff` 及对应预览段(regex 解析本地 TS 文件那一坨)。
  - 新增 `--empty-schema-db-url=<url>` flag + `DRIZZLEMAN_EMPTY_SCHEMA_DB_URL` 环境变量后备,缺失直接报错退出(在 preTarget 之前)。
  - Step A:introspect target DB → previewDir;复用既有的 `renameBaselineTag` 改名为 `0000_<slug>`。
  - Step B:`assertSchemaDbEmpty` 校验 schema DB。
  - Step C:`drizzle-kit push --dialect=… --schema=… --url=<schemaDbUrl> --force` 物化本地 schema 到 schema DB(`--force` 防 drizzle-kit 误判 data-loss 卡 prompt;空库场景实际只 CREATE)。
  - Step D:introspect schema DB → tmpDir(后缀 `.baseline-schemadbintro-<ts>/`)。
  - Step E:新增 `diffSnapshots()` 在两份 `meta/0000_snapshot.json` 上做结构化 diff,输出 `onlyInTarget` / `onlyInSchemaDb` 两组 `DiffSet`(tables / columns / indexes / foreignKeys / enums);entity key 形如 `table:public.users` / `column:public.users.id` / `index:public.users_email_idx` / `fk:public.users.users_owner_fk` / `enum:public.scan_status`。
  - Step F:新增 `chunkSql()` 把 introspect 输出的 SQL 切成 `{kind, key, sql, ownerTableKey?}` 三元组;支持 drizzle-kit 的 `--> statement-breakpoint` 分隔和 fallback 的「顶层 `;` 含括号 / 字符串 / 注释」扫描。
  - Step G:`buildDeltaSql` / `buildSchemaSql` 按差集筛三元组拼装两份 SQL;列差异在「该列所属表已整张被加进 0001/schema.sql 时」跳过(因整张 CREATE 已覆盖);单独缺列暂走 SQL chunker 抽不到的路径 → 留 feedback(下面记录)。
  - Step H:删 `tmpDir`;预览段重写为 3 条 artifact 摘要(路径 / size / 行数)+ schema.ts/relations.ts 提示;diff 摘要在 Step E 末尾打印。
  - Step I/J:decision prompt + apply 路径(backup → promote → reset DB)保留原逻辑,只把「DB backup 表 / fs backup 目录」的命名时间戳沿用。
  - `printSchemaDbReminder()`:每个终止路径(成功 / decline / non-TTY / error)都打印「schema DB url 已被填上本地 schema,请自行 drop」。
- **`README.md`** — baseline 表行刷新,反映新的必填 flag 与 4 份产物的语义。

### 偏差与遗留

- **缺列单独存在的 `ALTER TABLE ADD COLUMN` 暂未生成**:plan 关键点提到「列在两边都有的表里单独缺时,要从 snapshot 字段构造 `ALTER TABLE … ADD COLUMN`」。当前实现里 `chunkSql()` 不识别这种合成语句(SQL 抽块基于 introspect 原文,而 introspect 把列写在 `CREATE TABLE` 内联,没单独的 ALTER ADD COLUMN 可抽)。在实测的 `skai_v2_test` ↔ schema DB 场景下,`onlyInSchemaDb.columns=0` / `onlyInTarget.columns=0`,所以本轮端到端没有触发到这条路径。这条留作 follow-up(若以后真有 in-both-table-but-missing-column 的情况,需要新增「snapshot column → ALTER TABLE ADD COLUMN」的拼装器)— 已记到 `feedback.md`。
- **`schema.sql` 空场景未实测**:本次 diff 不空,占位分支未在端到端跑出。代码逻辑直观,未单独 mock 测;若需要可加单元测试。
